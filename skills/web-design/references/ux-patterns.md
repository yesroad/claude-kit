## 1. 첫인상 & 시각적 계층

**50ms 법칙**: 사용자의 미학적 판단은 50ms 이내에 완료되며, 이후 시간을 더 줘도 거의 바뀌지 않는다. (NN/G)

System 1(자동 판단)이 즉각 평가하는 3가지:
- **Information scent**: "이 정보가 내 목표와 관련 있는가?"
- **Credibility**: "이 사이트는 신뢰할 수 있는가?"
- **Aesthetics**: "이 사이트는 보기 좋은가?"

### 실무 적용

- 히어로 섹션에서 **누구를 위한 사이트인지 + 무엇을 제공하는지** 한눈에 파악 가능해야 함
- 악센트 색상은 **최대 1~2가지**로 제한 — 경쟁하는 강조 요소 금지
- 색상 4가지 이상 = "무지개 효과(rainbow effect)"로 시각적 혼돈 유발
- 빨간색 계열은 경고·오류 전용으로 남겨둘 것

### 시선 흐름 패턴

| 패턴 | 언제 발생하는가 | 대응 방법 |
|------|----------------|-----------|
| **F-패턴** | 텍스트 장벽, 흥미 부족 | 소제목·굵은 키워드 추가, 첫 2단락에 핵심 배치 |
| **레이어 케이크** | 소제목이 잘 된 콘텐츠 | 소제목 앞 2단어만으로 내용 파악 가능하게 작성 |
| **Z-패턴** | 비주얼 중심 랜딩페이지 | 좌상(로고) → 우상(nav) → 좌하(콘텐츠) → 우하(CTA) |

> **F-패턴은 나쁜 패턴이다.** 오른쪽·아래쪽 콘텐츠를 사용자가 놓치게 된다. 소제목과 글머리 기호로 레이어 케이크 패턴을 유도하라.

## 2. CTA 최적화

### 위치 원칙

- **"위치보다 맥락(context)"이 우선**이다. Above the fold라고 무조건 좋은 게 아님
- 신규 방문자에게 설명 없이 CTA를 먼저 보여주면 역효과 → **가치 제안 먼저, 행동 유도 나중**
- 긴 랜딩페이지는 **상·하단 이중 배치** 고려
- 히어로 이미지가 전체 화면을 채우면 스크롤 단서(chevron, 텍스트 등)를 반드시 추가할 것

### 색상 & 시각적 계층

- 버튼을 돋보이게 하는 건 특정 색상이 아니라 **주변과의 명도·채도 대비**
- 동일한 시각적 무게의 CTA가 2개 이상 → 사용자의 System 2(의식적 판단) 가동 → 전환율 하락
- 주요 CTA(filled) vs 보조 CTA(outlined/text)로 명확히 우선순위를 시각화할 것

### 카피 원칙

- "Get Started", "Learn More" 같은 모호한 레이블은 사용자를 오도한다 (NN/G 연구 증명)
- **"링크는 약속(a link is a promise)"** — 클릭 후 무슨 일이 일어나는지 정확히 알려줄 것
- 좋은 CTA 예시: "Take Our Style Quiz", "Shop Now for Back to School"

### 전환율 레이아웃 패턴

- **Trust signals(신뢰 요소)**는 CTA 인근에 배치 — 수상 경력, 미디어 인용, 사용자 리뷰
- **Value before ask**: 이메일·회원가입 요청 전에 가치를 먼저 경험하게 할 것
  - 로그인 장벽(login wall) 없이 먼저 기능 체험 → 그 후 계정 생성 유도

## 3. 네비게이션 패턴 (IA)

### 플랫폼별 기준 (NN/G 179명 정량 연구)

| 플랫폼 | 권장 패턴 | 근거 |
|--------|-----------|------|
| **데스크톱** | 상단 수평 바 또는 왼쪽 사이드바 | 햄버거 메뉴: 탐색 성공률 20%↓, 소요시간 39%↑ |
| **모바일 (4개 이하)** | 하단 탭바 또는 수평 링크 노출 | 모두 가시화 |
| **모바일 (5개 이상)** | 콤보 네비게이션 (중요 2~3개 고정 + 햄버거) | 햄버거 단독보다 15% 빠름 |
| **대시보드/문서** | 사이드바 (3단계 이상 계층) | 반전-L(상단+왼쪽)은 4단계까지 수용 |

> **데스크톱에서 햄버거 메뉴는 사용하지 않는다.** 화면 공간이 충분하면 모든 주요 항목을 노출하라.

### IA 설계 순서

1. **IA 초안 먼저** (콘텐츠 구조·관계 정의)
2. **네비게이션 UI는 나중에** (IA를 사용자에게 노출하는 방식)
3. 명칭(nomenclature)은 **내부 용어가 아닌 사용자 언어** 사용

### 모바일 vs 데스크톱 IA

**"모바일 퍼스트 ≠ 모바일 온리"** — 모바일 최적화를 데스크톱에 그대로 이식하지 말 것

| 모바일 | 데스크톱 |
|--------|----------|
| 프로그레시브 디스클로저 | 메가 메뉴 |
| 하단 탭 바, 스와이프 | 멀티 컬럼, 수평 탭 |
| 핵심 정보 최소 탭 도달 | 풍부한 호버 상태 활용 |

## 4. 폼 UX

### EAS 프레임워크 (NN/G 2025)

필드를 추가하기 전에 이 순서로 먼저 질문하라:

1. **Eliminate(제거)**: "이 필드가 꼭 필요한가?" 설명 못 하면 제거
   - 비긴급 정보는 나중 단계로 미룰 것
   - 조건 로직(conditional logic)으로 해당 없는 필드 숨기기
2. **Automate(자동화)**: 이미 아는 데이터는 재입력 요청하지 말 것
   - ZIP코드 → 시/도 자동완성, 카드번호 → 카드사 자동감지
3. **Simplify(단순화)**: 남은 필드는 스마트 기본값, 입력 마스크로 부담 최소화

### 필드 순서 원칙

친숙함(이름·이메일) → 우선순위(핵심 정보) → 의존성(이전 답변 기반) → 복잡성(쉬운 것) → 민감도(마지막)

- **단일 컬럼 레이아웃**이 원칙
- 멀티 컬럼은 City/State/Zip처럼 관련성 높고 짧은 필드에만 예외 허용

### 오류 메시지 규칙

| 항목 | 올바른 방식 |
|------|-------------|
| **타이밍** | 필드 완성 후 다음 필드로 이동한 시점 (blur 이후) |
| **조기 검증 금지** | 타이핑 시작 즉시 오류 = 적대적 패턴 |
| **인라인 검증** | 비밀번호 강도, 사용자명 중복만 허용 |
| **위치** | 해당 필드 바로 옆·아래 (상단 목록 단독 금지) |
| **내용** | 경고가 아닌 해결책 제안 ("대신 kim_park_01을 사용해보세요") |
| **표시 방식** | 색상 단독 금지 → 색상 + 테두리 + 굵은 텍스트 + 아이콘 조합 |

### Multi-Step Form 패턴

- **"one thing per page"** 원칙 — 각 화면에 하나의 질문 또는 관련된 소수 필드
- **프로그레스 인디케이터** 필수 — 전체 중 어디에 있는지 항상 표시
- **사전 공지** 필수 — 오래 걸리는 폼은 시작 전에 예상 시간, 필요 서류, 중간 저장 여부 안내
- 상호작용이 많은 단계(비교 선택)는 같은 화면에, 독립적인 단계만 분리

## 5. 인지 부하 원칙

### Hick's Law: 선택지 최소화

> "결정 시간은 선택지의 수와 복잡성에 비례한다."

- 결제 흐름, 온보딩, 긴급 상황에서는 선택지를 최소화
- 추천 옵션을 강조 표시(배지, 하이라이트)로 기본 결정 지원
- 신규 사용자에게는 **프로그레시브 온보딩**으로 기능을 점진적으로 소개
- 단, 지나친 단순화 = 사용자가 필요한 기능을 못 찾음 → 균형 필요

실무 사례: Google 홈페이지, Netflix 개인화 추천, Apple TV 리모컨

### Miller의 법칙: 청킹

> "작업 기억에는 7(±2)개 항목만 유지 가능하다."

- **7은 제한이 아니라 청킹의 중요성을 말하는 것** — "7개라서 OK" 오해 금지
- 콘텐츠를 의미 있는 단위(chunk)로 나눌 것
  - 전화번호: `010-1234-5678`
  - 폼: 배송 정보 그룹 / 결제 정보 그룹으로 분리
  - 네비게이션: 관련 카테고리로 그룹핑

### 프로그레시브 디스클로저

초기에는 핵심 옵션만, 사용자 요청 시 고급 기능 표시:
- **학습 용이성**, **사용 효율**, **오류율** 세 가지를 동시에 개선
- 3단계 이상 중첩은 사용자를 미로에 빠뜨릴 수 있으므로 피할 것
- Multi-step form은 스테이지드 디스클로저(선형적)의 구현 — 프로그레시브 디스클로저(계층적)와 다름

## 6. 마이크로카피

NN/G 2025 기준: 마이크로카피를 **목적 먼저** 정의하라.

| 목적 | 역할 | 예시 |
|------|------|------|
| **Informational** | 무엇을 해야 하는지 안내 | 제목, 소제목, 툴팁, 요약문 |
| **Influential** | 전환 유도 | CTA 버튼, 태그라인, 프로모션 배너 |
| **Interaction** | 인터페이스 조작 지원 | 버튼/링크 레이블, 플레이스홀더, 브레드크럼 |

- 각 텍스트는 **하나의 목적**만 가질 것 — 목적이 불분명하면 혼란 유발
- 좋은 CTA 카피: 즉각성 + 구체성 → "Shop Back-to-School Now!"
- 플레이스홀더는 레이블 대체 불가 — 입력 시 사라지면 사용자가 무엇을 입력했는지 잊음

## 7. UX 체크리스트

### 레이아웃 설계 시

- [ ] 히어로 섹션에서 50ms 안에 사이트 목적을 파악할 수 있는가?
- [ ] 악센트 색상이 1~2가지로 제한되어 있는가?
- [ ] 시선 흐름이 Z-패턴(비주얼 중심) 또는 레이어 케이크(텍스트 중심)로 유도되는가?
- [ ] 데스크톱에서 햄버거 메뉴를 사용하고 있지 않은가?

### CTA 설계 시

- [ ] 가치 제안 후 CTA를 배치했는가? (설명 먼저)
- [ ] 동일한 시각적 무게의 CTA가 2개 이상 없는가?
- [ ] CTA 카피가 클릭 후 일어날 일을 정확히 알려주는가?
- [ ] Trust signal이 CTA 인근에 배치되어 있는가?

### 폼 설계 시

- [ ] EAS 프레임워크로 불필요한 필드를 제거했는가?
- [ ] 단일 컬럼 레이아웃인가?
- [ ] 오류 메시지가 blur 이후에 표시되는가?
- [ ] 오류가 색상 + 테두리 + 아이콘으로 표시되는가?
- [ ] Multi-step form에 프로그레스 인디케이터가 있는가?

### 네비게이션 설계 시

- [ ] 모바일 4개 이하 항목은 모두 노출했는가?
- [ ] 모바일 5개 이상은 중요 항목 2~3개를 고정했는가?
- [ ] 명칭이 내부 용어가 아닌 사용자 언어인가?
