# PR 작성 가이드

> PR 본문 품질 기준 — `done.md`와 `git-operator.md`에서 참조

---

## 제목 형식

```
{type}({scope}): {한 줄 요약}
```

**규칙:**

- 50자 이내
- type은 영어 (`feat`, `fix`, `chore` 등)
- scope는 한글 또는 영어 명사 (애매하면 생략)
- 요약은 한글, 명령형 현재형 (`추가`, `수정`, `개선`)

**예시:**

```
feat(주문): 주문 목록 필터 옵션 추가
fix: 날짜 형식 검증 오류 수정
chore(의존성): TanStack Query v5 업그레이드
```

---

## 필수 섹션 템플릿

```markdown
## 작업 내용

- {변경 내용 bullet points}

## 디자인 (optional)

{디자인 URL 또는 스크린샷 (있는 경우)}

## 특이사항 (optional)

- {주의사항, 알려진 이슈, 추가 컨텍스트}
```

---

## 섹션별 작성 기준

### 작업 내용

| 기준 | 설명 |
|------|------|
| **필수** | 항상 포함 |
| **형식** | bullet point로 변경 내용 나열 |
| **수준** | 리뷰어가 코드를 보지 않아도 이해 가능한 수준 |
| **범위** | 이번 PR에서 변경된 것만 (이전 작업 설명 금지) |

```markdown
## 작업 내용

- 주문 목록 상단에 상태/날짜 필터 UI 추가
- 필터 변경 시 목록 재조회 로직 구현
- 선택된 필터 상태를 URL 파라미터로 유지
```

### 디자인

| 기준 | 설명 |
|------|------|
| **포함 시** | UI 변경이 있는 경우 |
| **생략 시** | 로직만 변경된 경우 |
| **형식** | Figma URL, 스크린샷, 목업 파일 경로 |

### 특이사항

| 기준 | 설명 |
|------|------|
| **포함 시** | 리뷰어가 알아야 할 맥락이 있을 때 |
| **예시** | "성능 이슈 있어 임시 처리", "다음 PR에서 리팩토링 예정" |

---

## 판단 기준표

| 변경 유형 | 작업 내용 | 디자인 | 특이사항 |
|-----------|-----------|--------|---------|
| 신규 기능 | 필수 | 있으면 포함 | 있으면 포함 |
| 버그 수정 | 필수 (원인 + 해결) | 생략 가능 | 재현 조건 있으면 포함 |
| 리팩토링 | 필수 (구조 변경 설명) | 생략 | 동작 변경 없음 명시 |
| 의존성 업데이트 | 필수 (변경 이유) | 생략 | 주요 변경사항 있으면 포함 |
| 문서 수정 | 필수 | 생략 | 생략 |

---

## 금지 사항

| 금지 | 이유 |
|------|------|
| 제목에 이모지 | 일관성 저하 |
| "작업했습니다", "수정했습니다" 형식 | 명령형으로 통일 |
| 빈 특이사항 섹션 유지 | 불필요한 노이즈 |
| 이전 PR 내용 포함 | 스코프 혼동 |
